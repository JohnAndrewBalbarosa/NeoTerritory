QUESTION:
Which OOP mechanism lets one interface call be served by different concrete implementations at runtime?

OPTIONS:
A. Encapsulation
B. Polymorphism
C. Macro expansion
D. Static linking

CORRECT ANSWER:
B. Polymorphism

RATIONALE:
Polymorphism (via virtual dispatch) lets a call through a base interface run different concrete behaviour. Encapsulation hides state; macros and linking are unrelated build-time mechanisms.

QUESTION:
What does the Singleton pattern guarantee?

OPTIONS:
A. A class has exactly one instance with a single global access point
B. A class can be freely subclassed and copied
C. Objects are produced in matching families
D. Objects are built through ordered steps

CORRECT ANSWER:
A. A class has exactly one instance with a single global access point

RATIONALE:
Singleton ensures a class has exactly one instance and provides one global access point to it. The other options describe subclassing, Abstract Factory, and Builder.

SOURCE REFERENCES:
* Gamma, Helm, Johnson, Vlissides (1994), Design Patterns

QUESTION:
How is the single-instance guarantee typically enforced in C++?

OPTIONS:
A. By marking every method const
B. By making the class abstract
C. By a non-public constructor plus a static accessor that returns the sole instance
D. By deleting the destructor

CORRECT ANSWER:
C. By a non-public constructor plus a static accessor that returns the sole instance

RATIONALE:
A hidden (non-public) constructor prevents outside construction, and a static accessor returns the one instance. The other options do not constrain instance count.

SOURCE REFERENCES:
* Nesteruk (2022), Design Patterns in Modern C++
* CodiNeo Creational module content
