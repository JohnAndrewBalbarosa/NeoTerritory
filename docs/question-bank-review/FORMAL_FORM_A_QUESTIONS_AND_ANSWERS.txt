QUESTION:
Which trio names the three classic Gang-of-Four pattern families?

OPTIONS:
A. Compiling, Linking, Running
B. Frontend, Backend, Database
C. Creational, Structural, Behavioural
D. Static, Dynamic, Virtual

CORRECT ANSWER:
C. Creational, Structural, Behavioural

RATIONALE:
The GoF families are Creational, Structural, and Behavioural. The other groupings are build phases, app tiers, and binding kinds — not pattern families.

SOURCE REFERENCES:
* Gamma, Helm, Johnson, Vlissides (1994), Design Patterns, ch. 1

B. Structural

CORRECT ANSWER:

Creational patterns address object creation/instantiation concerns. Structural deals with composition; Behavioural with communication; creation is squarely a Creational concern.

Which sentence correctly maps the three families by intent?

QUESTION:
Which OOP mechanism lets one interface call be served by different concrete implementations at runtime?

OPTIONS:
A. Encapsulation
B. Polymorphism
C. Macro expansion
D. Static linking

CORRECT ANSWER:
B. Polymorphism

RATIONALE:
Polymorphism (via virtual dispatch) lets a call through a base interface run different concrete behaviour. Encapsulation hides state; macros and linking are unrelated build-time mechanisms.

* CodiNeo Foundations module content

QUESTION:
Why do many design patterns depend on programming to an interface (abstraction) rather than a concrete class?

OPTIONS:
A. It lets implementations vary independently of the code that uses them
B. It always reduces the number of classes
C. It makes the program use less memory
D. It removes the need for constructors

CORRECT ANSWER:
A. It lets implementations vary independently of the code that uses them

RATIONALE:
Depending on an abstraction decouples clients from concrete types, so implementations can change/substitute without touching callers — the basis of many patterns. It does not inherently reduce classes/memory or remove constructors.

SOURCE REFERENCES:
* CodiNeo Foundations module content
* Gamma, Helm, Johnson, Vlissides (1994), Design Patterns, ch. 1

QUESTION:
Which statement best explains a limitation of patterns?

OPTIONS:
A. A pattern guarantees the design is correct for every problem
B. Patterns can only be used in object-oriented languages
C. Applying a pattern where it is not needed can add complexity without benefit
D. Patterns make code impossible to refactor later

CORRECT ANSWER:
C. Applying a pattern where it is not needed can add complexity without benefit

RATIONALE:
Patterns are solutions to specific recurring problems; forcing one where the problem is absent adds indirection and complexity for no gain. They are not universally correct, are not OO-exclusive, and do not block refactoring.

SOURCE REFERENCES:
* CodiNeo Foundations module content
* Gamma, Helm, Johnson, Vlissides (1994), Design Patterns, ch. 1
