QUESTION:
The Gang-of-Four catalogue groups patterns into three families. Which option lists them?

OPTIONS:
A. Behavioural, Structural, Creational
B. Public, Private, Protected
C. Stack, Heap, Register
D. Model, View, Controller

CORRECT ANSWER:
A. Behavioural, Structural, Creational

RATIONALE:
The three GoF families are Creational, Structural, and Behavioural (order aside). Access specifiers, memory regions, and MVC are unrelated groupings.

SOURCE REFERENCES:
* Gamma, Helm, Johnson, Vlissides (1994), Design Patterns, ch. 1

QUESTION:
A pattern primarily organises how responsibilities are assigned and how objects talk at runtime. Which family?

OPTIONS:
A. Creational
B. Structural
C. Behavioural
D. None — communication is not a pattern concern

CORRECT ANSWER:
C. Behavioural

RATIONALE:
Behavioural patterns concern responsibility assignment and runtime communication between objects. Creational is about making objects; Structural about composing them.

Which mapping of family-to-intent is correct?

OPTIONS:

B. All families address object creation only
C. Behavioural is a subset of Structural
D. Creational = object creation; Structural = composition/relationships; Behavioural = communication/responsibility

QUESTION:
A base-class pointer invokes a method and the derived class’s version runs. Which OOP concept is this?

OPTIONS:
A. Static binding
B. Encapsulation
C. Polymorphism
D. Preprocessing

CORRECT ANSWER:
C. Polymorphism

RATIONALE:
Dispatching to the derived override through a base pointer is runtime polymorphism. Static binding resolves at compile time; encapsulation and preprocessing are different concepts.

* CodiNeo Foundations module content

QUESTION:
What does depending on an abstraction (interface) buy a design that depending on a concrete class does not?

OPTIONS:
A. Guaranteed lower memory use
B. Substitutable implementations without changing client code
C. Fewer total source files in every case
D. Automatic thread safety

CORRECT ANSWER:
B. Substitutable implementations without changing client code

RATIONALE:
An interface dependency lets concrete implementations be swapped without editing clients — enabling extension and testing. It gives no memory, file-count, or thread-safety guarantee.

SOURCE REFERENCES:
* CodiNeo Foundations module content
* Gamma, Helm, Johnson, Vlissides (1994), Design Patterns, ch. 1

QUESTION:
Two engineers say “let’s use the Observer here.” What has the pattern name primarily given them?

OPTIONS:
A. A faster program at runtime
B. A guarantee the feature is bug-free
C. A way to skip writing the class entirely
D. A shared, compact way to communicate a known structure and intent

CORRECT ANSWER:
D. A shared, compact way to communicate a known structure and intent

RATIONALE:
The value is communication: the name conveys an agreed structure/intent so the team need not spell it out. It does not change runtime behaviour, remove bugs, or avoid implementation.

QUESTION:
When can adopting a design pattern be the wrong choice?

OPTIONS:
A. When the team already understands the pattern
B. When the underlying recurring problem the pattern solves is not actually present
C. Whenever the language supports inheritance
D. When the code must be unit tested

CORRECT ANSWER:
B. When the underlying recurring problem the pattern solves is not actually present

RATIONALE:
A pattern earns its complexity only when its target problem exists; applying it speculatively adds indirection without benefit. Team familiarity, inheritance support, and testing are not reasons to avoid it.

SOURCE REFERENCES:
* CodiNeo Foundations module content
* Gamma, Helm, Johnson, Vlissides (1994), Design Patterns, ch. 1
