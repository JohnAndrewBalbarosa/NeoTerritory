NeoTerritory Analyzer — Time & Space Complexity

n = input token count  |  m = AST nodes (O(n))  |  p = catalog patterns (constant, ~10)  |  s = symbols collected  |  k = candidate matches per site

Table 1. Per-stage time complexity

Pipeline stage Best case Average case Worst case Notes
Lexical analysis (tokenization)O(n)O(n)O(n)Single linear scan over source bytes.
Main tree / AST constructionO(n)O(n)O(n)One pass over the token stream; each token produces ≤1 node.
ReverseMerkle hashingO(m)O(m)O(m)Each node hashed exactly once; m = O(n).
Symbol table build (classes, functions)O(n)O(n)O(n)Hash-map inserts; amortized O(1) per symbol.
Symbol resolution / lookupO(1)O(1)O(s)Worst case is a degenerate hash collision chain.
Pattern matching (catalog × nodes)O(n)O(n · p)O(n · p)p is bounded by the catalog (~10), so effectively linear in n.
Pattern ranking (lexeme categories → structure tiebreak)O(1)O(k)O(k log k)k is the number of competing matches at one site (typically <5).
Subtree comparison (diffing)O(1)O(m)O(m)Best case: root hashes match → early exit. Worst case: full descent.
Output JSON generationO(f)O(f)O(f)f = findings + targets, bounded by O(n).
End-to-end pipelineO(n)O(n)O(n)Validated empirically: wall time ≈ 9.51·n + 1002 µs, R² = 0.9932 (n = 150 runs, AWS-production sweep).

Table 2. Per-stage space complexity

Pipeline stage Best case Average case Worst case Notes

Empirical calibration: AWS Lightsail production sweep, 7 input sizes × 5 repeats = 35 microservice samples, re-measured across 150 cohort runs. Slope = algorithmic cost per token; intercept = fixed per-invocation setup (process exec + catalog load), environment-dependent and not part of the algorithm.
