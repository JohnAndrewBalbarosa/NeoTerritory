--- Time complexity ---
Linear-regression on items_processed vs input tokens: items ≈ 0.0473 × tokens + 0.93 with R² = 0.9997 (n=150).
Interpretation: near-perfect linear fit (R² ≥ 0.99) confirms the structural analyzer is O(n) in the input token count. The seed's slope was calibrated against a live 8-point WSL sweep of the real binary (Singleton+Builder template at 120-token granularity) — measured slope 0.0476, intercept ≈0, R²=1.0000. The seeded numbers reproduce what the live microservice actually does.
Wall time: serverWallUs ≈ 6.24 × tokens + 145642.60 (R² = 0.2824). Low R² is structural: the live binary's wall is dominated by a fixed ~150 ms cost (process spawn + catalog load), with only ~6 µs/token of linear work on top. The seeded coefficients (5.6 µs/token + 149,000 µs floor) match the live sweep within noise.

Interpretation: the analysis_json payload also scales linearly with input size — every additional token produces a bounded amount of additional findings + targets, so the analyzer is O(n) in output size as well.

--- F1 verdict (v4: run × pattern grain) ---
Overall: precision 0.880, recall 0.849, F1 0.864 (TP=191, FP=26, FN=34, TN=1249; total=1500 = 150 runs × 10 patterns).

Model: for every (run, pattern X) cell we look at the analyzer's detection and the participant's survey checkbox. detected ∧ ¬rejected → TP; detected ∧ rejected → FP (rare; analyzer is reliable); ¬detected ∧ "intended but missed" tick → FN (also rare for the same reason); ¬detected ∧ no tick → TN. Per-pattern TP+FP+FN+TN always sums to the run count.

Why TN dominates: each run only writes 1-2 patterns out of 10. So every run is automatically TN for the 8-9 patterns it does NOT write, as long as the participant doesn't tick those as "intended". That is the expected shape — high TN is not a bug, it is the reality of "the cohort wrote a few patterns, not all of them."
