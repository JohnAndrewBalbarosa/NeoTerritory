# Docs guide

A one-page map of the `docs/` folder so you (or anyone landing fresh) can find the right reference without spelunking. Maintained alongside the docs themselves; update this file when adding a new top-level folder under `docs/`.

> **Source-of-truth contract.** Per `CLAUDE.md`, the markdown under `docs/Codebase/` IS the blueprint — every `Codebase/<path>/<file>.<ext>.md` describes a real source file at `Codebase/<path>/<file>.<ext>`. The code must mirror the docs 1:1, not the other way around. **If the code and the docs disagree, the docs win.**

## Folder-by-folder

### `docs/Codebase/` — the source-of-truth blueprint
Mirrors `Codebase/` one-to-one. Every folder + every file under `Codebase/` has a corresponding `.md` in `docs/Codebase/` describing intent, flow, collaborators, and key symbols. Generators / regenerators:

- `tools/generate_codebase_docs.ps1` walks **code → docs** (do NOT run this when docs and code disagree; the docs win, and this script would overwrite them).
- New design calls made during implementation → record in `docs/Codebase/DESIGN_DECISIONS.md` BEFORE writing dependent code.

Sub-trees:
- `docs/Codebase/Backend/` — `server.ts`, routes, middleware, db, services, AI pipeline.
- `docs/Codebase/Frontend/` — Vite SPAs (`index.html`, `admin.html`, `scraper.html`), components, hooks, marketing surfaces, studio.
- `docs/Codebase/Microservice/` — C++ binary, parser, pattern catalog, runtime introspection.
- `docs/Codebase/Infrastructure/` — Docker + deploy artefacts mirror.
- `docs/Codebase/Diagrams/` — architecture diagrams (PUML / Mermaid).
- `docs/Codebase/LegacyPatternTransformSamples/` — kept for historical reference.

### `docs/AUTH_SETUP.md`
End-to-end Google OAuth setup. Supabase project link, env vars (`AUTH_SUPABASE_SELF_HOSTED_URL`, `AUTH_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_KEY`), original-devs allowlist (`Codebase/Backend/src/config/originalDevs.ts`), and the role chooser flow at `/auth/choose`.

### `docs/DEPLOY.md`
AWS host topology, pm2 process names, the env-file location, the post-deploy smoke contract, and what `scripts/deploy.sh` does. Update whenever the AWS host changes shape (new env var, new pm2 process, new nginx route).

### `docs/SECURITY.md`
Threat surface + mitigations. JWT key handling, asymmetric-key sealed envelopes, RLS policies on the Supabase tables, secrets storage. Read this before adding any auth-touching code.

### `docs/ops/` — runbooks
One file per recurring operational issue. Add a new file when a flake/incident class appears for the second time. Currently:
- `aws-runc-flake.md` — Docker `runc` netns mount flake on the AWS host; symptoms, in-CI retry pattern, host-side `systemctl restart docker` recovery.

### `docs/auto/` — auto-generated
Regenerated by `tools/generate-ops-docs.mjs` and committed by the pre-commit hook (`.githooks/pre-commit`). **Do not edit by hand** — your changes will be clobbered next time someone touches a watched script. Includes:
- `ops-bash.md`, `ops-powershell.md` — reference for the bash + ps1 ops scripts.
- `scripts.md` — reference for `scripts/*.{sh,ps1,mjs}`.

### `docs/Research/` — bibliography
- `README.md` — index.
- `books/`, `papers/`, `industry-evidence/` — source materials backing the thesis claims.
- `codineo-thesis.md` — current thesis manuscript pointer.
- `testing-trophy.md` — the testing-pyramid analogue used by the project (static analysis + compile_run + unit_test phases).

### `docs/Questionnaire/`
Survey instruments (pre-test, per-run, end-of-session). `IMPLEMENTATION_PLAN.md` is the implementation roadmap; `REFERENCE.md` is the item bank.

### `docs/INFRA/`
Infrastructure migration notes and future-requirements tracking. `DOCKER_MIGRATION.md` captures the move to containerised builds; `FUTURE_REQUIREMENTS.md` is the running asks list.

### `docs/TODO/`
Per-domain TODO files. Granular tasks live here so the root `CHANGELOG.md` and `README.md` stay focused on shipped state.

## Hard rules touching `docs/`

These live in `CLAUDE.md` (the AI agent contract) but are worth knowing as a human contributor too:

1. **Docs win over code.** If the code and a doc disagree, regenerate or move the code to match the doc — never adjust the doc to match the code.
2. **One-to-one mirroring.** Every folder/file in `docs/Codebase/` corresponds to a real folder/file in `Codebase/` with the same name. No invented files.
3. **`DESIGN_DECISIONS.md` first.** New design calls made during implementation must be recorded BEFORE writing the dependent code.
4. **Auto-generated dirs are sacred.** Don't edit `docs/auto/` by hand — regenerate via `npm run docs:ops`.

## Adding a new top-level doc

1. Decide whether it belongs under an existing folder (`ops/`, `Codebase/`, etc.) or warrants a new one.
2. If new top-level: also add a row in this file's "Folder-by-folder" section.
3. If under `docs/Codebase/`: also add the matching real code file (or update the existing one) in the same commit, otherwise the 1:1 mirror is broken.
4. Cross-reference from `README.md` if it's something a new contributor should find immediately.
